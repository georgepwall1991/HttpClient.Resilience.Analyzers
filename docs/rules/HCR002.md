# HCR002

Long-lived manual `HttpClient` should configure `PooledConnectionLifetime`.

## Why

A static or otherwise long-lived manual `HttpClient` can keep connections alive longer than DNS changes or infrastructure rotation. Configure `SocketsHttpHandler.PooledConnectionLifetime` when manually owning the client.

## Bad

```csharp
private static readonly HttpClient Client = new();
```

```csharp
services.AddSingleton<GitHubClient>();

public sealed class GitHubClient
{
    private readonly HttpClient _client = new();
}
```

## Better

```csharp
private static readonly HttpClient Client = new(
    new SocketsHttpHandler
    {
        PooledConnectionLifetime = TimeSpan.FromMinutes(2)
    });
```

## Current Detection

The implementation reports static `HttpClient` fields and instance `HttpClient` fields on known singleton services when they are initialized with a default handler. Resolved custom types named `HttpClient` or `SocketsHttpHandler` are skipped. Qualified singleton registrations are matched against the declared containing type namespace so same-named services in other namespaces are skipped. It does not report ordinary instance fields, clients initialized with a `SocketsHttpHandler` object initializer that sets `PooledConnectionLifetime`, or clients initialized from a visible local or field handler whose initializer sets `PooledConnectionLifetime`.

## Code Fix

For simple parameterless `HttpClient` initializers, the code fix replaces the default initializer with a `SocketsHttpHandler` configured with `PooledConnectionLifetime = System.TimeSpan.FromMinutes(2)`. When the client already receives a handler argument, the analyzer reports the issue but does not offer an automatic rewrite because preserving custom handler configuration requires manual review.

## Suppression

Suppress only when DNS rotation is impossible or the handler lifetime is managed elsewhere and the analyzer cannot see it.
