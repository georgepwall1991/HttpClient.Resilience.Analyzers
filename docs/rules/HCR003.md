# HCR003

Do not cache `IHttpClientFactory.CreateClient()` results long-term.

## Why

Factory-created clients are intended to be requested when needed while the factory manages handler lifetime underneath. Caching those clients in static fields or known singleton objects can bypass the intended lifetime model and reintroduce stale-client behavior.

## Bad

```csharp
public sealed class ClientCache
{
    private static HttpClient _client = null!;

    public static void Initialize(IHttpClientFactory factory)
    {
        _client = factory.CreateClient("github");
    }
}
```

```csharp
public sealed class ClientCache
{
    private static readonly HttpClient Client = Factory.CreateClient("github");
}
```

## Better

```csharp
public sealed class ClientRunner(IHttpClientFactory factory)
{
    public async Task RunAsync(CancellationToken cancellationToken)
    {
        var client = factory.CreateClient("github");
        using var response = await client.GetAsync("/repos", cancellationToken);
    }
}
```

## Current Detection

The implementation reports assignments or field initializers from visible `IHttpClientFactory.CreateClient(...)` calls into static fields, into instance fields on types whose names end with `Singleton`, and into instance fields on services known from compilation-wide `IServiceCollection`-shaped `AddSingleton<TService>()` or `AddSingleton<TService, TImplementation>()` registrations. Factory receivers are matched as simple `IHttpClientFactory` or explicitly qualified `System.Net.Http.IHttpClientFactory`, qualified singleton registrations are matched against the declared containing type namespace so same-named services in other namespaces are skipped, and lookalike `CreateClient(...)` methods on other factory types are skipped.

## Suppression

Suppress only when the analyzer cannot see a custom lifetime boundary and the client is not actually cached long-term.
